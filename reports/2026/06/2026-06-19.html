RAG · Architecture Contextual Retrieval: Replace Naive RAG Chunking With Anthropic's Contextual Embedding Pattern

The #1 failure mode of RAG systems in production: chunks lose meaning when separated from their document. Anthropic's Contextual Retrieval pattern (now production-proven at scale) fixes this by prepending each chunk with a model-generated context summary before embedding. The workflow: (1) Split document into standard chunks (512-1024 tokens). (2) For each chunk, call a fast model with: "Given the full document, write a 2-sentence context for this chunk: {chunk}". (3) Concatenate context + chunk, then embed the combined text. The result: retrieval recall improves 30-50% because the embedding now captures document-level semantics, not just local text. For iOS apps with on-device RAG (using Core AI + vector stores): generate contextual prefixes at indexing time on your server, ship the pre-contextualized embeddings to the device. The cost is ~2x embedding tokens at index time, but zero additional cost at query time. Combine with hybrid search (BM25 keyword + vector similarity) for best results. Anthropic RAG Docs →

Tool Use · Patterns The "Tool Sandwich" Pattern: Guarantee Structured Output From AI Agents Every Time

When building agentic workflows that call tools, the biggest reliability issue is malformed tool calls — the model generates invalid JSON, hallucinates parameter names, or calls tools in the wrong order. The "Tool Sandwich" pattern eliminates this: (1) Pre-tool reasoning — force the model to output a structured plan in a <thinking> block before any tool call. (2) Tool execution — validate the call against your schema before executing; reject and retry on malformation. (3) Post-tool reflection — after the tool returns, force the model to summarize what it learned before proceeding. Implementation in Claude Code / Codex: wrap your system prompt with explicit tool_choice: "required" for steps that must call tools, and use tool_choice: "none" for reflection steps. For iOS agents using Function Calling: define strict JSON schemas with additionalProperties: false and enable Structured Outputs mode — this guarantees valid JSON every call, at the cost of ~5% higher latency. The payoff: agent success rate jumps from ~70% to 95%+ on multi-step tasks.

Agent Arch · MCP Design Your Agent's Tool Surface Like an API: The Principle of Minimal Authority

The most common anti-pattern in agent architectures: giving the agent access to every tool in your system "just in case." This creates security risks, increases hallucination surface, and wastes context tokens on tool descriptions the agent never uses. The fix: apply the Principle of Minimal Authority (PoLA) — give each agent only the tools it needs for its current task scope. Implementation pattern for MCP servers: (1) Define tool profiles — named sets of tools for specific workflows (e.g., "code-review" profile only exposes read tools, never write). (2) Rotate tools per conversation turn based on the current phase. (3) Use inputSchema constraints to limit what values tools accept (e.g., a file-write tool that only accepts paths within /src/). For production agent deployments at Binance: create a tool gateway that wraps your MCP servers with per-agent allow-lists, rate limits, and audit logging. Every tool invocation should produce a structured log entry: {agent_id, tool_name, params, result_hash, timestamp}. This is the minimum governance layer for any agent touching production systems.
