RAG · Architecture Contextual Retrieval: How to Build RAG That Actually Works for Code-Heavy Domains

Standard RAG pipelines fail on codebases because naive chunking splits functions mid-body and embedding models can't capture structural relationships. The proven pattern for code-aware RAG: (1) Use AST-aware chunking — parse files into functions/classes/modules rather than fixed-size text chunks. For Swift, use SwiftSyntax to split by declaration boundaries. (2) Generate contextual headers for each chunk: prepend the file path, parent class name, and a one-line AI-generated summary before embedding. Anthropic's research shows this reduces retrieval failures by 49% vs. raw chunking. (3) Implement hybrid retrieval: combine vector similarity (semantic) with BM25 keyword search (exact matches on function names, error codes). For iOS codebases: index your project with contextual headers like // File: Sources/Networking/APIClient.swift | Class: APIClient | Purpose: Handles all REST API requests with retry logic before each chunk. Tool Use · Patterns Structured Tool Use: Design Agent Tools That Succeed 95% of the Time

The #1 reason AI agents fail in production isn't model capability — it's poorly designed tool interfaces. After observing thousands of agent sessions, the pattern that maximizes tool-call success: (1) Narrow tool scope — each tool does exactly one thing. Instead of a generic database_query tool, create specific get_user_by_id, list_recent_orders, search_products tools. Models pick the right tool 95%+ of the time when names are self-explanatory. (2) Exhaustive parameter descriptions — include valid ranges, enum values, and example values in the schema. (3) Return structured errors — when a tool fails, return {"error": "user_not_found", "suggestion": "try search_users instead"} so the agent can self-correct. (4) Include a "think" tool — a no-op tool the agent can call to reason about complex decisions before acting. For iOS MCP servers: expose your app's capabilities as granular tools — get_crash_reports(last_hours=24), check_feature_flag(name="new_trading_ui"), get_app_metrics(metric="p95_latency"). The agent orchestrates them into diagnostic workflows automatically.

Agent Arch · Multi-Agent The Supervisor-Worker Pattern: Scale Agent Reliability With Hierarchical Delegation

Single-agent architectures hit a ceiling around 15-20 tool calls — beyond that, context pollution causes degraded reasoning. The supervisor-worker pattern solves this: a lightweight "supervisor" agent decomposes tasks and delegates subtasks to specialized "worker" agents, each with their own fresh context window. Architecture: (1) Supervisor receives the user request, creates a plan (3-7 steps), and dispatches each step to the appropriate worker. (2) Workers are specialized — one for code generation, one for testing, one for documentation, one for deployment. Each worker has a focused system prompt and limited tool access. (3) Handoff protocol — workers return structured results; the supervisor validates before proceeding. If a worker fails, the supervisor can retry with modified instructions or escalate. For production: implement this in Claude Code using subagents (Task.spawn), or with OpenAI's Agents SDK using handoff patterns. The key insight: workers don't need frontier-model intelligence — use Haiku/GPT-4o-mini for routine workers, reserve Opus/GPT-5.5 for the supervisor's planning step.
