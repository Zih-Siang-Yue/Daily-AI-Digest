Tool Use · Agent Design Multi-Tool Agent Architecture: Design Patterns for Reliable AI Agents

The biggest failure mode in AI agents isn't bad models — it's bad tool design. The 2026 consensus on building reliable multi-tool agents: (1) Typed tool schemas: define every tool with strict JSON Schema parameters — the model should never need to guess parameter formats. In Swift: struct SearchTool: AgentTool { let parameters: SearchParams }. (2) Tool routing via description quality: the agent selects tools based on their descriptions, not names. Write descriptions like documentation: "Searches the user's transaction history for amounts, dates, and counterparties. Returns max 20 results sorted by recency." (3) Fallback chains: if tool A fails (API timeout), automatically try tool B (cached data) before erroring. (4) Result validation: every tool call returns a ToolResult with a confidence score — the orchestrator only passes high-confidence results to the synthesizer. (5) Rate limiting per tool: prevent runaway loops where the agent calls the same API 50 times. For iOS agent architectures: wrap your Binance API endpoints as typed tools with rate limits, define clear fallback paths (live API → cached response → error message), and use TaskGroup for parallel tool execution. Prompt Engineering · System Design The "GhostApproval" Lesson: Securing Agent Workflows Against Prompt Injection

This week's GhostApproval vulnerability — which affected 6 major AI coding assistants by tricking agents into accessing files outside their sandbox via symbolic links — reveals a universal prompt engineering lesson: human-in-the-loop approval is not security. The defense patterns for any agent system: (1) Input sanitization at the tool level: before executing any file operation, resolve all symlinks and verify the canonical path is within the allowed scope: let resolved = URL(fileURLWithPath: path).resolvingSymlinksInPath(); guard resolved.path.hasPrefix(sandboxRoot) else { throw .accessDenied }. (2) Principle of least privilege: each tool gets only the permissions it needs — a "read code" tool cannot write files, a "search" tool cannot execute commands. (3) Structured output validation: never pass raw agent output to system calls. Parse into typed structs first, then execute. (4) Audit logging: every tool call is logged with input/output/timestamp — enables post-incident forensics. (5) Sandbox isolation: run agent tool calls in a separate process with restricted entitlements (on iOS, use NSFileCoordinator with explicit scoped access). The meta-lesson: treat AI agent actions with the same security posture as untrusted user input — validate, sanitize, restrict, log. OWASP LLM Top 10 →

GhostApproval · Security

"GhostApproval" Vulnerability Hits 6 Major AI Coding Assistants

Security researchers disclosed GhostApproval, a vulnerability affecting at least six major AI coding assistants (Amazon, Anthropic, Google products). The flaw exploits symbolic link handling to trick AI agents into accessing and modifying sensitive files outside their workspace — effectively bypassing human approval prompts. The attack: a malicious repo contains symlinks that resolve to sensitive paths; when the AI agent processes the repo, it follows symlinks and gains unintended access. For developers: (1) Never clone untrusted repos into your AI workspace. (2) Add .cursor/rules or CLAUDE.md entries: "Never follow symbolic links outside the project root."
