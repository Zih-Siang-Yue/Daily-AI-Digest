RAG · Context Engineering Context Engineering > Prompt Engineering: The 2026 Mental Model Shift

The industry is converging on a reframing: what we called "prompt engineering" in 2024 is now better described as context engineering — the art of assembling the right information (context) for a model to reason over, not just writing clever instructions. The practical difference: a perfect prompt with bad context produces garbage; a simple prompt with excellent context produces gold. The RAG architecture pattern for 2026: (1) Hybrid retrieval: combine semantic search (embeddings) with keyword search (BM25) using Reciprocal Rank Fusion. Neither alone is sufficient — semantic misses exact terms, keyword misses meaning. Use score = 1/(k + rank_semantic) + 1/(k + rank_keyword) with k=60. (2) Contextual compression: retrieved chunks often contain 70% noise. Before inserting into the prompt, run a lightweight model (GPT-5.6 Luna or Gemini Flash) to extract only the relevant sentences. This cuts token costs 3x while improving answer quality. (3) Dynamic context windows: don't use fixed chunk sizes. Split documents at semantic boundaries (paragraph breaks, section headers), then retrieve variable-length chunks that preserve reasoning coherence. (4) Context freshness signals: timestamp every retrieved chunk and include [Retrieved: 2 hours ago] or [Retrieved: 6 months ago] in the context — models use this to weight information appropriateness. For iOS: if you're building an AI-powered help system or document Q&A in your app, this RAG stack runs beautifully on-device with MLX embeddings + Foundation Models for generation. Tool Use · Function Calling Tool Use Architecture: Design Your Agent's API Surface Like a Senior Engineer

The quality of AI agent output depends heavily on how you design the tools it can call. Poorly designed tool schemas create ambiguity; well-designed schemas create precision. The patterns that produce the best results in 2026: (1) Single-responsibility tools: each tool does exactly one thing. Don't create manage_database(action: "create" | "read" | "update" | "delete") — create four separate tools. The model routes more accurately to specific tools than to action parameters within one tool. (2) Rich parameter descriptions: the description field in your JSON Schema is the model's primary guide. Include: what valid values look like, what happens on invalid input, and one concrete example. Compare: "query": {"description": "SQL query"} (bad) vs. "query": {"description": "A SELECT statement against the trades table. Must include WHERE clause limiting date range. Example: SELECT * FROM trades WHERE created_at > '2026-07-01'"} (good). (3) Error envelopes: tool responses should return structured errors, not raw exceptions. Return {"success": false, "error_code": "RATE_LIMITED", "retry_after_ms": 2000, "suggestion": "Reduce batch size or add delay"} — the model can then self-correct. (4) Idempotency tokens: for any tool that mutates state, require an idempotency_key parameter. When the model retries (which it does frequently), duplicate calls are safely ignored. For iOS developers building MCP servers or App Intents: these principles apply directly — your AppIntent parameter descriptions ARE the model's understanding of your app's capabilities.
