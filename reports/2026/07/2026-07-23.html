Quantization · 4-Bit Standard The 2026 On-Device Deployment Recipe: Quantize to 4-Bit, Target Neural Engine, Ship

On-device model deployment has settled into a clear standard recipe in 2026 — and if you're not following it, you're leaving performance on the table. The steps: (1) Start with a 7B or smaller model — MLX community has proven that 7B models (Llama 3.3, Mistral 7B, Phi-4) quantized to 4-bit deliver GPT-4-level quality for domain-specific tasks after fine-tuning. (2) Quantize with GPTQ or AWQ to 4-bit: use mlx-lm quantize --model your-finetuned-model --bits 4 — this reduces memory footprint 4× and memory traffic 4× per generated token. A 7B model goes from 14GB (FP16) to 3.5GB (4-bit). (3) Convert to CoreML with coremltools: target the Neural Engine compute unit — this is critical. If you leave the default (CPU/GPU), you waste the most efficient accelerator on Apple Silicon. Use ct.convert(model, compute_units=ct.ComputeUnit.ALL) and let the runtime dispatch optimally. (4) Fine-tune with QLoRA for your domain: instead of deploying a generic model, fine-tune a LoRA adapter (typically 50-100MB) on your specific use case (trading terminology, Binance-specific patterns) and deploy the adapter separately. This means OTA updates are tiny — you're updating a 50MB adapter, not a 3.5GB base model. (5) Measure memory bandwidth, not TOPS: the key insight of 2026 — for autoregressive generation, memory bandwidth is the bottleneck. ExecuTorch · Cross-Platform Edge ExecuTorch 2.0: Meta's Cross-Platform Edge AI Framework Hits Production Maturity

ExecuTorch 2.0 (Meta's successor to PyTorch Mobile) has reached production maturity in 2026, offering the most complete cross-platform edge AI deployment story outside Apple's ecosystem. When to use ExecuTorch vs CoreML: CoreML is optimal when targeting Apple hardware exclusively (best Neural Engine utilization). ExecuTorch wins when you need (a) cross-platform deployment (iOS + Android from a single model pipeline), (b) custom operator support beyond what CoreML handles, or (c) models trained in PyTorch that don't convert cleanly to CoreML. Practical setup for iOS: (1) Install via SPM: .package(url: "https://github.com/pytorch/executorch", from: "2.0.0"). (2) Export your model: torch.export.export(model, args) | to_edge() | to_executorch() — the pipeline automatically applies quantization and delegates operations to Apple's Neural Engine via CoreML backend or Metal backend. (3) Key feature — dynamic shapes: ExecuTorch 2.0 supports dynamic input shapes natively, meaning your model can handle variable-length sequences (chat history, varying document lengths) without padding tricks. CoreML requires fixed shapes or enumerated shape sets. (4) Benchmark tip: compare ExecuTorch's Metal delegate vs CoreML export for your specific model — in many cases CoreML still wins by 15-20% on Apple Silicon due to tighter hardware integration, but ExecuTorch offers more deployment flexibility. Trending The Memory Bandwidth Wall: Why On-Device AI Performance Is Actually About RAM, Not Chips

A critical insight crystallizing in 2026's edge AI discourse: memory bandwidth is the true bottleneck for on-device generative AI, not compute power (TOPS/TFLOPS). Here's why this matters for your architectural decisions: autoregressive token generation requires reading the full model weights for every single token produced. At 4-bit quantization, a 7B model is ~3.5GB. Generating one token means reading 3.5GB from memory. At iPhone 16 Pro's ~100 GB/s memory bandwidth, that's ~35ms per token read — setting a hard ceiling of ~28 tokens/sec regardless of Neural Engine compute power. The implication for iOS developers: (1) Model selection should prioritize parameter efficiency over capability — a well-fine-tuned 3B model that fits in L2 cache will outperform a 7B model that's bandwidth-limited, for your specific task. (2) Speculative decoding becomes essential — use a tiny draft model (300M params, cached in SRAM) to predict likely next tokens, then verify batches against the larger model. This amortizes the bandwidth cost across multiple tokens. (3) Apple's unified memory architecture is the edge AI advantage — ARM CPUs with separate CPU/GPU memory can't compete with Apple Silicon's unified pool because every data copy between memory spaces adds latency. (4) Prompt caching is non-negotiable — for repeated inference patterns (same system prompt, similar user queries), cache the KV state to avoid recomputing from scratch. This single optimization can 3-4x your apparent throughput for conversational interfaces. Profile with Metal System Trace to find your specific bottleneck before optimizing.
