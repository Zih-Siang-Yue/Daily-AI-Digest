Agentic RAG · Architecture Agentic RAG: Move Beyond Naive Retrieval with Self-Correcting Multi-Step Pipelines

The "naive RAG" pattern (embed → retrieve → generate) is now considered a prototype-only approach in 2026. Production systems demand Agentic RAG — a multi-stage pipeline where an LLM agent orchestrates retrieval, evaluates quality, and self-corrects before generating output. Here's the architecture for your iOS app's knowledge features: (1) Query Transformation: before hitting your vector DB, route the user query through a lightweight model that rewrites it for optimal retrieval — expanding abbreviations, resolving pronouns, and generating sub-queries for complex questions. Use claude-haiku or gpt-5.6-luna for this step ($0.10/MTok). (2) Adaptive Retrieval Router: a classifier decides the retrieval strategy per query — simple factual lookups go to keyword search (BM25), semantic questions use vector similarity, and multi-hop reasoning triggers GraphRAG across your knowledge graph. (3) Re-ranking with Cross-Encoders: after initial retrieval returns 20-50 chunks, a cross-encoder model (like Cohere Rerank or a fine-tuned BERT) re-scores and selects the top 5-8 most relevant passages. This single step improves answer quality by 30-40% in benchmarks. (4) Self-Correction Loop (CRAG): the generator produces an answer, then a verifier checks it against the retrieved context. If confidence is below threshold, the agent triggers a second retrieval pass with a refined query. Implement this as a while confidence < 0.85 && attempts < 3 loop. (5) For iOS integration: run the query transformer and re-ranker on-device using CoreML models (both are small enough at ~100MB), with only the generation step hitting a cloud API. This gives you sub-200ms retrieval with cloud-quality generation. Anthropic RAG Guide →

Tool Use · Best Practices Tool Use Prompt Engineering: Make Your Agent's Function Calls 3x More Reliable

If your AI agent's tool calls are unreliable (wrong parameters, calling the wrong tool, or not calling tools when it should), the problem is almost always in your tool descriptions, not the model. Here's the 2026 best-practice framework for tool-use prompt engineering: (1) Rich descriptions with examples: don't just write "Fetches user data" — write "Fetches a user's profile data including name, email, and preferences. Call this when the user asks about their account, settings, or personal information. Example: user says 'what's my email?' → call get_user_profile(user_id=current_user)". Models route to tools based on description similarity to the user's intent. (2) Enum parameters over free-text: wherever possible, use z.enum(["buy", "sell", "hold"]) instead of z.string(). Constrained parameter spaces eliminate entire categories of errors. (3) Required vs. optional clarity: mark parameters as required only when the tool genuinely fails without them. Over-constraining causes the agent to hallucinate values rather than ask the user. (4) Negative examples in system prompt: add a section like "NEVER call place_order without explicit user confirmation. NEVER call delete_account from a general query about settings." Negative constraints prevent the most dangerous failure modes. (5) Tool grouping by domain: organize your MCP server's tools into logical groups (market_data, portfolio, account) — models handle 30+ tools well when they're semantically grouped, but struggle with 30 flat unrelated tools. (6) Test with adversarial prompts: specifically test ambiguous queries that could trigger multiple tools ("show me my balance" — wallet balance or account balance?) and ensure the agent asks for clarification rather than guessing. Agent Arch · Multi-Agent Patterns The Supervisor-Worker Pattern: Build Reliable Multi-Agent Systems for Production

Multi-agent systems (like Codex's new Multi-Agent V2, Cursor's parallel agents, or your own custom setups) need a reliable coordination pattern. The Supervisor-Worker architecture has emerged as the production standard in 2026: (1) Supervisor Agent: a single orchestrator that receives the user's high-level goal, decomposes it into tasks, assigns each task to a specialized worker, and aggregates results. The supervisor uses a frontier model (Sonnet 5, GPT-5.6 Terra) for its planning capability. (2) Worker Agents: specialized, narrow agents — each handles one task type (code generation, test writing, documentation, code review). Workers use cheaper models (Luna, Flash) since their scope is constrained. (3) Shared Memory: a central state store (Redis, or even a shared file in simpler setups) where workers read task assignments and write results. The supervisor polls this store to track progress. (4) Human-in-the-Loop Gates: the supervisor pauses for human approval at defined checkpoints — after the plan is generated (before execution), after critical file modifications, and before any destructive operations. (5) Error Recovery: if a worker fails, the supervisor can (a) retry with a different model, (b) decompose the task further, or (c) escalate to the human. Never let a failed worker block the entire pipeline. Implementation for iOS teams: use this pattern for your CI/CD AI pipeline — a supervisor agent receives "prepare release 3.2.1" and orchestrates: Worker A runs tests, Worker B generates changelog from git log, Worker C updates version strings, Worker D creates the release branch. Each worker is a simple, testable unit; the supervisor handles coordination. LangGraph Multi-Agent Guide →

Trending Adaptive RAG: The Query-Aware Architecture Replacing Static Retrieval Pipelines

The hottest infrastructure pattern in July 2026 is Adaptive RAG — systems that dynamically choose their retrieval strategy based on query complexity, rather than running every question through the same pipeline. The architecture: a lightweight classifier (can be rule-based or a small model) evaluates each incoming query and routes it to one of several retrieval paths: (1) No-retrieval path: for questions the model can answer from parametric knowledge (common knowledge, simple reasoning). Skip retrieval entirely — saves latency and cost. (2) Single-hop retrieval: standard vector search for factual lookups ("what's our API rate limit?" → search docs → answer). (3) Multi-hop retrieval: for questions requiring information synthesis across multiple documents ("how does our auth system interact with rate limiting during peak load?"). Uses iterative retrieval with query reformulation between hops. (4) GraphRAG path: for relationship queries ("which services depend on the user service?") — leverages a knowledge graph rather than flat document search. Why this outperforms static RAG: benchmarks show 25-35% improvement in answer quality AND 40-60% reduction in retrieval latency because simple queries skip expensive retrieval entirely. For iOS apps with AI features: implement this pattern by adding a QueryRouter that inspects the question before choosing a retrieval strategy. Even a heuristic router (check if the query contains entity names → use retrieval, otherwise → use parametric) dramatically improves both speed and quality. Adaptive RAG Paper →
